# Smoke E2E — Sessão 1.4.B (Visitor↔Buyer Matching + UTM Stitcher 2.0)

> Roteiro manual de validação end-to-end das 3 estratégias de matching + UTM Stitcher 2.0.
> Tempo estimado: **20 minutos** (5 min setup + 15 min cenários).

## Pré-requisitos

- 1.4.A já validado — tracking script ingere eventos e popula `tracking_visitors`
- Webhook gateway (Hotmart sandbox ou Generic) já mapeado pra workspace
- App rodando: `pnpm dev`
- Trigger.dev rodando: `pnpm exec trigger.dev@latest dev` (paralelo)
- 1+ campanha em `campaigns` com `name = 'Black Friday 2026'` (pra perfect match)

## Cenário 1: Deterministic xcode (confidence 1.0)

**Setup:**

1. Abra página instrumentada com `criation-tracking.js` instalado
2. Abra DevTools console: `criation('debug')` → confirma `visitor_id` populado em cookie `_cio_vid`
3. Copie o `visitor_id` do output debug
4. Clique em link `data-criation-checkout` ou navegue pro checkout Hotmart — script faz link enrichment injetando `xcode=<visitor_id>` no link
5. Confirme na URL final do checkout: `?...&xcode=<visitor_id>...`

**Trigger:** 6. Complete a compra no checkout sandbox (ou simule webhook Hotmart com `tracking.external_code = <visitor_id>`)

**Validação SQL:**

```sql
SELECT
  id, external_code, matched_visitor_id, visitor_match_strategy,
  visitor_match_confidence, visitor_matched_at
FROM gateway_events
WHERE id = '<gateway_event_id_recente>';
-- Esperado:
--   visitor_match_strategy = 'deterministic_xcode'
--   visitor_match_confidence = 1.0000
--   matched_visitor_id = <visitor_id>
--   visitor_matched_at = <timestamp recente>

SELECT identified_buyer_email_hash, identified_at
FROM tracking_visitors WHERE visitor_id = '<visitor_id>';
-- Esperado: identified_buyer_email_hash populado (hash do email da compra)

SELECT count(*) FROM tracking_events
WHERE visitor_id = '<visitor_id>' AND matched_buyer_email_hash IS NOT NULL;
-- Esperado: > 0 (eventos do visitor receberam o hash retroativamente)
```

## Cenário 5: Reverse matching com OVERRIDE de unmatched (audit A2)

**Setup:** cliente compra primeiro (matcher direto roda → marca `unmatched`), depois acessa lead magnet onde identify roda.

1. Complete compra Hotmart com email `comprador@teste.com` (vai gerar `gateway_events` SEM externalCode, fbclid ou utm_campaign matchável)
2. Aguarde matcher rodar — task `stitch-gateway-event` marca `visitor_matched_at = now()` e `visitor_match_strategy = 'unmatched'`
3. Em outro browser/aba (visitor diferente), acesse landing instrumentada
4. Console: `criation('identify', 'comprador@teste.com')`
5. Aguarde task `process-tracking-event` rodar reverse matching

**Validação SQL** (use `pgcrypto` digest — `sha256()` puro não existe em PG):

```sql
-- O gateway_event do passo 1 deve ter sido SOBRESCRITO pelo reverse:
SELECT visitor_match_strategy, matched_visitor_id, visitor_match_confidence,
       visitor_matched_at, match_strategy, stitched_at
FROM gateway_events
WHERE customer_email_hash = encode(digest('comprador@teste.com', 'sha256'), 'hex');
-- Esperado:
--   visitor_match_strategy = 'reverse_email'  (audit A3 — strategy correta, não 'clickid')
--   visitor_match_confidence = 0.8500
--   matched_visitor_id = <visitor_id do passo 3>
--   visitor_matched_at = <timestamp do passo 4>
--   match_strategy = 'visitor' OU 'unmatched' (depende se UTMs do visitor resolveram campaign)
--   stitched_at = <timestamp recente> (audit B3 — re-stitch disparado)
```

**Validação tracking_subscriptions (audit B4):**

```sql
-- Se a compra era subscription, identified_visitor_id deve estar populado:
SELECT identified_visitor_id FROM gateway_subscriptions
WHERE subscriber_code = '<subscriber_code_da_compra>';
-- Esperado: <visitor_id do passo 3>
```

## Cenário 7: Cascata de precedência

Garanta que estratégias mais fortes ganham:

- xcode UUID v4 válido + fbclid no mesmo gateway → deve resolver `deterministic_xcode` (xcode ganha)
- xcode UUID v4 inexistente + fbclid válido → deve resolver `clickid` (xcode falha, cai pro próximo). Bonus: log warn de cross-workspace deve aparecer (audit C7)
- xcode NÃO-UUID (ex: CPF "123.456.789-00") → estratégia ignorada silenciosamente (audit B5 — aceitar qualquer string como confidence 1.0 era bug)
- Manual mapping em `utm_mappings` para a UTM da venda → `match_strategy = 'manual'` (manual ganha sobre visitor)
