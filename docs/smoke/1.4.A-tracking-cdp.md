# Smoke E2E — Sessão 1.4.A (Criation CDP)

> Roteiro manual de validação end-to-end do tracking script + ingestion endpoint.
> Tempo estimado: **10 minutos** (5 min setup + 5 min testes).

## Pré-requisitos

- App rodando local: `pnpm dev`
- Usuário logado com workspace ativo
- Conexão Supabase válida (variáveis em `.env.local`)
- Conta Trigger.dev configurada — para a task `process-tracking-event` rodar, execute:
  ```bash
  pnpm exec trigger.dev@latest dev
  ```
  (em outro terminal, paralelo ao `pnpm dev`)

## 1. Setup inicial (1 min)

1. Abra `http://localhost:3000/configuracoes/conexoes`
2. Verifique que o grupo **"Tracking & analytics"** aparece com card `Tracking Criation` em status `Não conectado`
3. Clique no card → dialog abre mostrando 0 eventos
4. Feche dialog. Clique em **"Gerenciar"** (ou navegue direto pra `/configuracoes/tracking-script`)
5. Verifique:
   - Status block mostra "Aguardando 1º evento"
   - Snippet block mostra `<script async src="..." data-workspace="<seu-uuid>"></script>`
   - Origens permitidas: vazio (modo onboarding)
   - Eventos recentes: empty state

## 2. Página de teste (2 min)

Crie `tmp/test-tracking.html` na raiz do projeto:

```html
<!doctype html>
<html lang="pt-BR">
  <head>
    <meta charset="utf-8" />
    <title>Teste CDP Criation</title>
    <!-- Cole aqui o snippet da página /configuracoes/tracking-script -->
    <script
      async
      src="http://localhost:3000/criation-tracking.js"
      data-workspace="SEU-WORKSPACE-UUID"
    ></script>
  </head>
  <body>
    <h1>Smoke test — CDP Criation</h1>
    <button data-criation-event="hero_cta" data-criation-position="top">CTA Hero</button>

  </body>

```

Abra via `file://` ou `python3 -m http.server` em http://localhost:8080.

## 3. Validações (5 min)

### 3.1 page_view automático

1. Abra a página com UTMs:
   `http://localhost:8080/test-tracking.html?utm_source=fb&utm_campaign=smoke_test&fbclid=ABC123`
2. Abra DevTools → Network → filtra por `track`
3. Verifique:
   - ✅ Request POST `http://localhost:3000/api/v1/track`
   - ✅ Status `204 No Content`
   - ✅ Latência < 200ms (idealmente < 100ms)
   - ✅ Request body (Payload tab) contém: `event_name: "page_view"`, `utms.utm_source: "fb"`, `fbclid: "ABC123"`, `visitor_id` UUIDv4

### 3.2 Cookies first-party

1. DevTools → Application → Cookies → `localhost:8080`
2. Verifique:
   - ✅ `_cio_vid` (UUIDv4, expira em ~90d)
   - ✅ `_cio_utms` (JSON com UTMs persistidos)

### 3.3 Persistência no DB

1. Rode no Supabase SQL Editor (ou `mcp__supabase__execute_sql`):
   ```sql
   SELECT event_name, visitor_id, event_ts, utms, fbclid, client_ip_hash, page_url
   FROM tracking_events
   WHERE workspace_id = 'SEU-WORKSPACE-UUID'
   ORDER BY event_ts DESC
   LIMIT 5;
   ```
2. Verifique:
   - ✅ Row aparece em < 2s após o page_view
   - ✅ `event_name = 'page_view'`
   - ✅ `utms->>'utm_source' = 'fb'`
   - ✅ `fbclid = 'ABC123'`
   - ✅ `client_ip_hash` populado (não-null)
   - ✅ `page_url` contém `test-tracking.html`

### 3.4 Visitor upsert (Trigger.dev async)

1. Aguarde 3-5s
2. Rode:
   ```sql
   SELECT visitor_id, first_utm_source, last_utm_source, first_click_id, total_events
   FROM tracking_visitors
   WHERE workspace_id = 'SEU-WORKSPACE-UUID'
   ORDER BY last_seen_at DESC LIMIT 5;
   ```
3. Verifique:
   - ✅ Row aparece (task `process-tracking-event` rodou)
   - ✅ `first_utm_source = 'fb'`
   - ✅ `first_click_id = 'ABC123'` (fbclid)
   - ✅ `total_events >= 1`

### 3.5 Auto-events (scroll + click + form)

1. Role a página até 50% → no Network: novo POST com `event_name: "scroll"`, `custom_data.milestone: 50`
2. Clique no botão "CTA Hero" → POST com `event_name: "hero_cta"`, `custom_data.position: "top"`
3. Submeta o form → POST com `event_name: "form_submit"`, `custom_data.form_name`

### 3.6 Identify

1. No DevTools Console:
   ```js
   window.criation('identify', 'smoke-test@example.com')
   ```
2. Network → novo POST com `event_name: "identify"`, `identify_email: "smoke-test@example.com"`
3. SQL:
   ```sql
   SELECT identified_buyer_email_hash, identified_at
   FROM tracking_visitors
   WHERE workspace_id = 'SEU-WORKSPACE-UUID' AND identified_buyer_email_hash IS NOT NULL;
   ```
4. ✅ Hash SHA-256 do email aparece

### 3.7 Track custom

```js
window.criation('track', 'video_play', { video_id: 'vsl_v1', value: 497 })
```

- ✅ POST com `event_name: "video_play"`, `custom_data: { video_id: "vsl_v1", value: 497 }`

### 3.8 Link enrichment (retrocompat 1.4.8)

1. Inspecione o `<a href>` do "Comprar via Hotmart"
2. ✅ href agora tem `?utm_source=fb&utm_campaign=smoke_test&fbclid=ABC123` injetado

### 3.9 UI live (status + eventos recentes)

1. Volte pra `http://localhost:3000/configuracoes/tracking-script`
2. Refresh
3. Verifique:
   - ✅ Status mudou para "Recebendo eventos" (verde)
   - ✅ Eventos 24h > 0
   - ✅ Tabela "Eventos recentes" lista os disparados (page_view, scroll, hero_cta, form_submit, identify, video_play)

### 3.10 Origins allowlist

1. Em `/configuracoes/tracking-script`, adicione origem inválida `not--valid_`
2. ✅ Mostra erro "Formato inválido"
3. Adicione `localhost` (válido)
4. ✅ Aparece na lista
5. Hard reload da página de teste — agora postar de `localhost:8080` em vez de `localhost`:
   - Se origin allowlist é `localhost`, mas página é `localhost:8080` → match via "bare host acerta subdomain" → ✅ passa
   - Adicione `*.cliente.com` e teste de domínio external — deve rejeitar com 403 `origin_not_allowed`

### 3.11 CORS preflight

```bash
curl -i -X OPTIONS http://localhost:3000/api/v1/track \
  -H "Origin: https://cliente.com" \
  -H "Access-Control-Request-Method: POST" \
  -H "Access-Control-Request-Headers: Content-Type"
```

- ✅ 204 + headers `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: POST, OPTIONS`

### 3.12 Rate limit

Em DevTools console, dispara burst:

```js
for (let i = 0; i < 700; i++) window.criation('track', 'spam', { i })
```

- ✅ Primeiros ~600 retornam 204
- ✅ Excesso retorna `429 rate_limited`

### 3.13 Dedup defensivo

1. Pegue um payload do Network (Copy as fetch)
2. Reenvie 3x via curl — exatamente o mesmo `event_id` + `event_ts`
3. SQL: `SELECT COUNT(*) FROM tracking_events WHERE event_id = 'xxx'`
4. ✅ COUNT = 1 (UNIQUE absorve retries)

## Notas

- **Trigger.dev offline:** Se a task não roda (Trigger.dev dev server não está up), visitor upsert não acontece, mas evento persiste OK. Validação 3.4 falha sem isso.
- **Latência > 200ms:** Possíveis causas — Supabase região distante (deveria ser sa-east-1), pool maxout, query lenta. Investigar via `pg_stat_statements`.
- **403 falso positivo:** Se origin allowlist estiver populada e teste estiver em domínio fora — esperado. Remover allowlist ou adicionar `localhost`.
