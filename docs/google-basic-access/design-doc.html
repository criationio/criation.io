Submitted with the application for Basic access to the Google Ads Developer Token · Owner contact: me@heywhispa.com · Document version 2026-05-15

Company Name

Criation.io

Business Model

Criation.io is a B2B SaaS for advertisers and marketing agencies. We help our customers improve return on ad spend by unifying the first-party conversion data captured on their own websites and providing AI-assisted funnel and attribution diagnostics on top of that data. Customers subscribe to one of two monthly plans (Pro and Agency); each plan is bundled with a credit allowance that the customer spends on analyses, and pricing scales with credit usage. We do not buy media, do not manage ad campaigns on behalf of our customers, and do not resell Google Ads inventory.

Tool Access / Use

The Criation.io product is accessed by our customers' own marketing, growth and analytics teams through a multi-tenant web application (workspace per customer). Each customer connects their own Google Ads account to their Criation workspace via a one-click Google OAuth flow; the OAuth grant is per customer and is initiated and revocable by the customer at any time. Internally, Criation.io employees have read-only operational access through admin tools (token visibility is restricted; all admin actions are recorded in an audit log).

Once connected, the customer uses our setup wizard to (1) pick which Google Ads account they want Criation.io to send conversion data to and (2) map the events captured on their website (for example, purchase, lead, add_to_cart) to the conversion actions that exist in the selected Google Ads account. We then run a server-side fanout job that forwards each qualifying event from the customer's website into the Google Ads account they connected. The customer can toggle a test mode (server-side validateOnly: true) to verify the payload structure without logging the conversion against their ad performance.

Two automated jobs run on our infrastructure on the customer's behalf:

  • a token-refresh cron, daily, that rotates the customer's OAuth access token using the long-lived refresh token they granted us;
  • a conversion fanout job, near real-time (~10 minute pickup window), that ingests new conversion events from the customer's website into the Google Ads conversion actions they selected.

Tool Design

The conversion data flow is one-directional, from the customer's website into the customer's own Google Ads account. Criation.io does not read back any campaign, ad, keyword, budget, bidding or reporting data from Google Ads; the only reads we perform are the metadata needed to populate the setup wizard (list of accessible accounts, list of conversion actions per account).

API Services Called

Google Ads API (requires the developer token under application)

Used exclusively to populate the setup wizard so the customer can choose which Google Ads account and which conversion action to wire up. Read-only, called from server-side code authenticated with the customer's OAuth access token under the https://www.googleapis.com/auth/adwords scope.

Endpoint Purpose
GET /v24/customers:listAccessibleCustomers List the customer IDs that the OAuthed Google user can access, so the wizard can show them as connectable accounts. (docs)
POST /v24/customers/{id}/googleAds:searchStream
SELECT customer_client.id, customer_client.descriptive_name, customer_client.currency_code, customer_client.time_zone, customer_client.test_account, customer_client.manager FROM customer_client WHERE customer_client.status = 'ENABLED' 		For an MCC (manager) customer, list the sub-accounts that are actually billable Google Ads accounts. We only call this against the customer's own MCC and only when the customer has manager permissions on that account.
POST /v24/customers/{id}/googleAds:searchStream
SELECT conversion_action.id, conversion_action.resource_name, conversion_action.name, conversion_action.type, conversion_action.category FROM conversion_action WHERE conversion_action.status = 'ENABLED' 		For a non-manager account that the customer wants to wire up, list the conversion actions defined in that account so the wizard can present them as targets.

Google Data Manager API (separate; does not require the developer token)

Used for the actual conversion fanout. The Data Manager API does not consume the Google Ads developer token; it is listed here only for completeness of the integration design.

Endpoint Purpose
POST https://datamanager.googleapis.com/v1/events:ingest Forward hashed first-party conversion events captured on the customer's website into the conversion actions the customer selected in the wizard. Supports validateOnly: true for test mode, and transactionId for deduplication with the customer's existing client-side Google tag. (docs)

What we do NOT do

  • We do not create, edit, pause or delete campaigns, ad groups, ads, keywords or assets.
  • We do not read or modify budgets, bidding strategies or bid adjustments.
  • We do not pull campaign performance reports or any reporting metric back from Google Ads.
  • We do not call the Google Ads API on behalf of customers we do not represent (each call is scoped to a specific workspace and uses the OAuth token granted by the owner of that workspace).
  • We do not redistribute, resell or share Google Ads data with any third party.

OAuth scopes requested

  • https://www.googleapis.com/auth/adwords — required to call the three Google Ads API endpoints listed above.
  • https://www.googleapis.com/auth/datamanager — required to call the Data Manager API.
  • https://www.googleapis.com/auth/cloud-platform — required by the Data Manager API per official setup guide.
  • openid, email, profile — used only to display the connected Google account's name and email back to the customer in the wizard.

Data handling & security

  • PII hashing. All user-identifying fields (email, phone, first name, last name) are SHA-256 hashed before being sent to the Data Manager API, in line with Google's Enhanced Conversions requirements.
  • Token storage. OAuth access and refresh tokens are encrypted at rest with AES-256-GCM (versioned key). Tokens never leave our backend, are not exposed to the client, and are scrubbed from application logs (path-based redaction).
  • Per-customer isolation. All persistence is gated by row-level security keyed on workspace; an OAuth token granted by customer A can never be used to query customer B's account.
  • Rate limit posture. We perform metadata reads once per customer at OAuth callback time and refresh them on a low-frequency cadence; we do not poll Google Ads API for reporting. The conversion fanout is bounded per workspace via worker concurrency limits and backs off on RESOURCE_EXHAUSTED.
